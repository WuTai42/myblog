但是实际上也可以不用完全匹配，只要向顶点着色器提供了输入签名的数据即可，也就是说，顶点数据可以有一些顶点着色器根本用不到的的数据

输入签名定义的是：在执行顶点着色器的出程序过程中，将数据从输入寄存器中读取时所视作的类型

如果两个格式不匹配会发出警告，但是这不一定是错误，有可能是程序员故意而为止。 （这样做的后果？？）

1. 创建常量缓冲区

常量缓冲区也是一种GPU资源（ID3D12Resource）,其数据可供着色器程序所引用。

前面经常出现的代码

常量缓冲区对象 cbuffer 名称为 cbPerObject。内部储存的是一个4x4的矩阵 gWorldViewProj，表示把一个点从局部空间变换到齐次剪裁空间所用的由世界、视图、和投影矩阵3种变幻组合而成的矩阵。

常量缓冲区和顶点缓冲区与索引缓冲区不同的地方是，常量缓冲区通常由CPU每帧更新。

当摄像机每帧不停移动时，常量缓冲区也需要在每一帧都随之以新的视图矩阵而更新。

所以我们要将常量缓冲区创建到一个上传堆而非默认堆中，这样做能使我们从CPU端更新常量。

常量缓冲区对硬件也有特别的要求。常量缓冲区的大小必须为硬件最小发分配空间（256B）的整数倍。

2. 更新常量缓冲区

常量缓冲区是由 D3D12_HEAP_TYPE_UPLOAD 这种堆类型来创建的，所以我们可以通过 CPU 为常量缓冲区资源更新数据

但是首先我们要通过map 方法获取到指向欲更新数据资源的指针

参数1： 子资源的索引（后面会再讲解，这里默认置为0）

参数2： 指向 D3D12_RANGE 结构体的指针，该指针描述了内存的映射范围（可选项），置为空指针就是对整个资源进行映射。

参数3： 利用双重指针，返回待映射资源数据的目标内存块，通过使用 memcpy 函数将数据从系统内存复制到常量缓冲区

在常量缓冲区更新完成之后，我们应该在释放映射内存之前对其进行 Unmap （取消映射）操作

tips: map 和 umap 是多线程安全的函数，而且还可以嵌套使用

5. 根签名和描述符表

我们应该将不同的着色器程序所需的各种类型资源绑定到渲染流水线上，实际上，不同类型的资源会绑定到特定的寄存器槽上（register slot），以供着色器程序访问。

根签名：在执行绘制命令之前，那些应用程序将绑定到渲染流水线上的资源，会被映射到着色器对应的输入寄存器。在创建流水线的时候会对其进行验证，不同的绘制调用可能会用到一组不同的着色器程序，也就意味着要用到不同的根签名。

D3D中根签名由ID3D12RootSignature接口来表示，并且以一组描述绘制调用过程中着色器所需资源的根参数定义而成。

根参数可以是根常量、根描述符或者描述符表。

7. 编译着色器

在D3D中，着色器程序必须被编译为可移植的字节码。之后图形驱动程序将获取这些字节码，并将其重新编译为针对当前系统GPU所优化的本地指令[ATI1]。

我们可以在运行期间用下列函数对着色器进行编译。

pFileName : 我们希望编译的以.hlsl 作为扩展民的HLSL源代码文件

pDefines : 不使用，置为 nullptr

pInclude : 不使用，置为 nullptr

pEntryPoint : 着色器的入口点函数名。一个 .hlsl 文件可能存有多个着色器程序，所以我们需要为待编译的着色器指定入口点

pTarget : 指定所用着色器类型和版本的字符串。

Flags : 指定对着色器代码应当如何编译的标志，我们常用的两种是 D3DCOMPILE_DEBUG 和 D3DCOMPILE_SKIP_OPTIMIZATION 。

Flags2 : 一种高级选项，暂时不用

ppCode : 返回一个指向 ID3DBlob 数据结构的指针，存储着编译好的着色器对象字节码

ppErrorMsgs : 返回一个指向 ID3DBlob 数据结构的指针，发生编译错误的时候自动存储报错的字符串

9、流水线状态对象

大多数控制图形流水线状态的对象被统称为**流水线状态对象(Pipeline State Object, PSO)**，用 ID3D12PipelineState 接口来表示。

要创建 PSO 要先填写一份描述其细节的 D3D12_GRAPHICS_PIPELINE_STATE_DESC 结构体实例。

在实例填写完成之后，便可以调用 ID3D12Device::CreateGraphicsPipelineState 方法来创建 ID3D12PipelineState 对象。

ID3D12PipelineState 对象集合了大量的流水线状态，为了保证性能，我们将这些所有的对象集合到一起，一并送到渲染流水线去。

由于PSO的验证和创建操作都特别的费时，所以我们尽量要在初始化阶段就生成PSO。在用完之后就将其存于类似散列表这样的集合里面，以便以后的快速读取。

并非所有的渲染状态都封装与PSO内，比如 viewport 和 剪裁矩形 等独立于PSO。

Direct3D 实际上就是一种状态机，里面的事物会保持它们各自的状态，直到我们将其改变。
